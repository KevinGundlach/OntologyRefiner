We are building a set of LLM agents for the purpose of extracting information on metal alloy corrosion testing from a given set of research papers in materials science. 

Our ultimate goal is to construct a self-supervised agentic framework that compiles a detailed list of all variables deemed important for predicting the pitting potential of a metal alloy in a given test solution under a given test environment. 

In each paper there is a list of "base materials" - the metal alloys that are studied. The variables for each "base material" will include things like chemical composition, type of alloy (e.g., austenitic stainless steel, duplex, ferritic, martensitic, Ni-base superalloy), and microstructure (e.g., fully recrystallized, sigma-phase precipitates, elongated grains, martensitic).

For each "base material" we have one or more "conditioned materials" which are the result of any kind of processes/alterations applied to the base material. The variables for each "conditioned material" will include things like manufacturing process, heat treatments, and chemical alterations that could affect its corrosion pitting potential. 

The Pessall & Nurminen paper, for example, identifies "W5" as a base material, where "W502", "W582", and "W549" are the conditioned materials resulting from the application of different heat treatments to "W5".

For each "conditioned material" we have one or more "experiments". Each experiment represents a single complete procedure for measuring the pitting potentials of a conditioned material in a particular test solution in a particular environment. The variables for each "experiment" will include things like the test solution's chemical composition, ph value, and temperature.

Since we are unaware of precisely which variables are relevant, the agentic system needs to be self-refining. The system consists of four agents - a "Data Extractor", a "Critic", a "Consolidator", and a "Refiner".

1. Data Extractor

The data extractor's job is to populate a markdown document with the requested information, given a materials science research paper like one of the attached examples. The document structure is as follows:

# Base Material

    * identifier: If the authors provided a name for this base material, list it here.
    
    * alloy composition: Extract the alloy's precise chemical composition. Please mention all the chemical elements used, including any trace elements.

    * alloy type: Please provide a brief paragraph describing the type of alloy. For example, whether it's an austenitic steel, whether it's a duplex, a stainless, a ferritic, martensitic, etc...

    * microstructure: Provide a brief paragraph describing the alloy's microstructure. For example, whether it has a pearlite martensitic microstructure.

## Conditioned Material

    * identifier: If the authors provided a name for this conditioned material, list it here.

    * process description: Provide a brief paragraph describing how the material was prepared, manufactured, or processed. 

    * heat treatment: Provide a brief paragraph describing the heat treatments applied to this alloy.
    
### Experiment

    * test procedure: Provide a full account of the test procedure used on this conditioned material. Was potentiostatic polarization used? Was potentiodyanmic polarization used? What was the scan rate? Provide a full summary of the entire procedure.

    * test solution and environment: Provide all information about the test solution and environment, including details such as the electrolyte used, the chloride ion concentration, the solution pH, the solution temperature, and any other information regarding the test solution and environment mentioned in the paper.  

    * pitting potential: Extract values of the pitting potential measured for this conditioned material following this test procedure. Provide values and units of meausure.

There can be one or more "Experiment" sections per "Conditioned Material" section, and there can be one or more "Conditioned Material" sections per "Base Material" section. 

The data extractor's output is in simple plain-text markdown so that it can focus all its attention on extracting accurate comprehensive data, rather than on trying to fit it to a strict complex schema.

2. Critic

Notice that the list of variables given to the data extractor above is incomplete. For example, I've omitted information regarding the reference electrode, surface finish, and sample geometry. Notice also that in requesting the base material's alloy composition, we failed to specify that we require the units of measurement (weight %, atomic %, ppm, etc...).

The Critic's job is to analyze the current paper, the Data Extractor's given markdown template, and the Data Extractor's output. The Critic will then produce a list of any missed variables (along with their definitions) and, if needed, refined definitions of existing variables to be more precise and complete. The critic's output does not need to differentiate between newly proposed variables vs. existing refined variables. 

The Critic's output should thus be a JSON object with the following structure:

{
    "proposed_base_material_variables": [
        {"name": "example_variable_1", "definition": "example definition of variable 1"},
        {"name": "example_variable_2", "definition": "example definition of variable 2"},
        {"name": "example_variable_3", "definition": "example definition of variable 3"},
        etc...
    ],

    "proposed_conditioned_material_variables": [
        {"name": "example_variable_4", "definition": "example definition of variable 4"},
        {"name": "example_variable_5", "definition": "example definition of variable 5"},
        {"name": "example_variable_6", "definition": "example definition of variable 6"},
        etc...
    ],

    "proposed_experiment_variables": [
        {"name": "example_variable_7", "definition": "example definition of variable 7"},
        {"name": "example_variable_8", "definition": "example definition of variable 8"},
        {"name": "example_variable_9", "definition": "example definition of variable 9"},
        etc...
    ],
}


3. Consolidator.

The critic may recommend many unimportant extraneous 'edge case' variables, or repeatedly recommend the same variables when given different papers but with different names and slightly different definitions. 

After the data extractor & critic have been run on a batch of papers (perhaps 10 or 20), the consolidator will run. It will be given a set of variables & corresponding definitions, consisting of those the data extractor is requesting along with those the critic recommends. That is, the input will have the form: 

[
    {"name": "example_variable_1", "definition": "example definition of variable 1"},
    {"name": "example_variable_2", "definition": "example definition of variable 2"},
    {"name": "example_variable_3", "definition": "example definition of variable 3"},
    etc...
]

There shouldn't be a need to distinguish between base material variables, conditioned material variables, and environment variables in the consolidator's input. Rather, the consolidator will be run separately on each group of variables so that there's no "cross contamination".

The consolidator will then output a JSON object having the following form. It will associate a "normalized name" to every variable defined in the given list (to identify duplicates), and also associate a normalized definition for every normalized variable (to account for slight nuances / slight differences in the critic's generated definitions for duplicates).

Thus, the output of the consolidator will be a JSON object having the form: 

{
    "normalized_variable_mapping": [
        {"original_name": "example_variable_1", "normalized_name": "normalized_variable_1"},
        {"original_name": "example_variable_2", "normalized_name": "normalized_variable_1"},
        {"original_name": "example_variable_3", "normalized_name": "normalized_variable_2"},
        ...
    ],
    
    "normalized_variable_definitions": [
        {"name": "normalized_variable_1", "definition": "definition of normalized variable 1},
        {"name": "normalized_variable_2", "definition": "definition of normalized variable 2},
        ...
    ]
}

A human will then review the complete list of normalized variable names and definitions. A python script will count the number of times the critic recommended each variable across the batch, using the consolidator's output to match up the critic's output with the normalized variable names. This should correspond to the number of papers that mentioned each variable, and so can be used to estimate each variable's importance.

Any variables deemed important will be manually added to the appropriate sections of the data extractor's template. The extractor->critic->consolidator loop will then be run again on another batch of 10-20 papers.


4. Refiner

When all batches of papers have been processed through the Data Extractor, Critic, and Consolidator, the data extractor's template will likely contain some variables whose definitions are overly broad. In the above example Data Extractor template, we mentioned "test solution and environment" as one variable having a complex comprehensive definition. Many of the details, such as such as the chemical composition of the test solution, the ph value, the temperature, etc... should be split into separate variables rather than aggregated into a single one.

The agentic framework may then be run again on the full collection of papers, beginning with this new refined data extractor template. It is our expectation to see the Data Extractor template converge to a single comprehensive list of all variables deemed relevant to a metal alloy's pitting potential. 


The purpose of this agentic framework is to construct a collection of knowledge graphs - one for each experiment - to serve as the dataset on which to train a graph neural network to predict corrosion pitting potential. There are about 100 research papers in total. These papers are provided in Markdown format.

Please provide me the prompts to use for all four agents - the Data Extractor, the Critic, the Consolidator, and the Refiner. 

Also, are there any shortcomings with this approach? Are there any critical details that I've failed to account for in this agentic refinement loop? Any pitfalls I may run into?
