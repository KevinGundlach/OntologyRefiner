### 1. Data Extractor Prompt

**System/Instruction Prompt:**
> You are an expert Materials Scientist and Data Extraction Agent. Your task is to analyze materials science research papers and extract precise data regarding metal alloy corrosion testing, specifically focusing on factors that influence the pitting potential.
>
> You will be provided with a research paper (which may include text, tables, and figures). Your goal is to populate the hierarchical Markdown template below. 
>
> **Extraction Rules:**
> 1.  **Be Exhaustive:** Scan all text, tables, and figures. If exact numbers are not in the text, carefully estimate them from relevant plots if possible, but explicitly state when a value is an estimate.
> 2.  **Handle Missing Data:** If a requested variable is entirely missing, output "Not specified in text." Do not invent or hallucinate data.
> 3.  **Traceability:** Append a brief location tag (e.g., `[Page 3, Table 2]` or `[Section: Experimental Method]`) to your extracted values so human reviewers can verify them.
> 4.  **Hierarchy:** A paper may have multiple Base Materials. A Base Material may have multiple Conditioned Materials (including "As-received"). A Conditioned Material may have multiple Experiments. Repeat the Markdown sections as necessary to capture all relationships accurately.
>
> **Output Template:**
> # Base Material
> * identifier: [Name or ID of the base material]
> * alloy composition: [Precise chemical composition, including trace elements and units of measure, e.g., wt%, at%]
> * alloy type: [Brief paragraph describing the alloy classification]
> * microstructure: [Brief paragraph describing the microstructure]
> 
> ## Conditioned Material
> * identifier: [Name or ID of the conditioned material]
> * process description: [Brief paragraph describing preparation, manufacturing, or processing]
> * heat treatment: [Brief paragraph describing applied heat treatments]
> 
> ### Experiment
> * test procedure: [Full summary of the test procedure, e.g., potentiostatic polarization, scan rate]
> * test solution and environment: [All details including electrolyte, chloride ion concentration, pH, temperature, deaeration, etc.]
> * pitting potential: [Extracted values with units of measure]

### Analysis: Shortcomings & Pitfalls

Your framework is robust, but there are a few critical details and technical realities you should account for before scaling this to 100 papers.

**1. The "Estimating from Plots" Pitfall**
You noted that the Data Extractor should "estimate values from a given plot." While multimodal models can process images, extracting precise numeric values from complex, multi-line scientific plots (especially those with logarithmic scales or overlapping data points) is highly prone to hallucination. 
* **Correction:** Treat LLM plot estimation as a last resort. If the data is only in a graph, instruct the Extractor to output: "Data only available in Figure X; exact numeric value not stated." You can then use specialized plot-digitizer software (like WebPlotDigitizer) to extract the data manually, which is standard practice for systematic reviews.

**2. The Traceability Gap**
When you manually review the extracted data or the Critic's proposed variables, you need to know exactly *where* the model found the information. If the model says the pitting potential is 420 mV, finding that in a 15-page double-column PDF is tedious. 
* **Correction:** I added a "Traceability" rule to the Extractor prompt above. Force the model to append the page number, section, or table number to its extractions. For example, in the Pessall & Nurminen paper you provided, the model should output `Conditioned Material: W502 [Page 3, Table 4]`. 

**3. "As-Received" Hierarchy Issues**
Your hierarchy forces a `Base Material -> Conditioned Material -> Experiment` flow. However, many papers test the base material in its raw, "as-received" state without any conditioning. 
* **Correction:** Explicitly instruct the Extractor that "As-received" or "Unconditioned" is a valid entry under the `Conditioned Material` tier. Otherwise, the model might skip the Conditioned Material block entirely, breaking your parsing script, or hallucinate a conditioning step.

**4. Consolidator Context Collapse**
The Consolidator is only receiving a list of names and definitions. Without the context of the original papers, it might make erroneous groupings. For instance, "annealing time" and "exposure time" might get merged into "time" if the Critic wrote overly brief definitions for both.
* **Correction:** Ensure the Critic is prompted to write highly specific definitions (as instructed in the updated prompt), and review the Consolidator's output carefully before updating your schema.

**5. Critic Over-Sensitivity (Schema Bloat)**
The Critic is incentivized to find *something* missing. If you run it on 100 papers, it might propose 400 highly niche variables (e.g., "beaker volume," "stirring bar RPM"). 
* **Correction:** Your planned Python script to count variable frequencies is the perfect defense against this. Set a strict threshold—for instance, only add a variable to the master schema if it is recommended in at least 15% of the batch.
